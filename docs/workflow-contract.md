# Workflow Contract

Este documento define o ciclo de desenvolvimento do Legion. Ele nao e fonte de status atual.
Para estado vivo, pendencias e proxima acao, leia
`docs/HANDOFF.md`.

## Objetivo

Permitir automacao forte sem perder decisao humana onde ela e necessaria.
O fluxo oficial e:

```text
intake -> classified -> planned -> approved -> executing -> verifying -> human-review -> closing -> learned
```

## Estados Canonicos

| Estado | Entrada | Saida | Responsavel primario |
|---|---|---|---|
| `intake` | pedido do Andre ou evento de sessao | tarefa entendida e contexto minimo lido | `start` / agente atual |
| `classified` | escopo inicial | nivel alpha/beta/gamma e risco inicial | `plan` |
| `planned` | nivel definido | plano, PRD ou spec com criterio de aceite | `plan` |
| `approved` | plano com human gate | aprovacao humana registrada ou gate dispensado | Andre |
| `executing` | escopo aprovado | diff, artefato ou mudanca operacional | skill de dominio |
| `verifying` | diff/artefato pronto | evidencia objetiva PASS/WARN/FAIL/ABSTAIN | `qa` / Auditor |
| `human-review` | risco alto, ABSTAIN ou decisao critica | decisao humana explicita | Andre |
| `blocked` | human-review resolveu como "blocked" | tarefa parada; entrada manual | Andre |
| `closing` | entrega verificada | handoff, vault, commit/push quando cabivel | `end` |
| `learned` | feedback/resultado | aprendizado persistido no vault/contexto | `end` / `memory` |

### Estado `blocked` (off-path)

`blocked` e o estado de saida quando `human-review` decide que o trabalho nao pode prosseguir.
Uma tarefa em `blocked` requer intervencao manual: o Andre deve re-planejar ou encerrar a tarefa.
O state machine aceita `blocked` como estado valido, mas ele nao possui saida canonica automatica;
a transicao de volta deve ser feita manualmente para `planned` (se re-planejada) ou `closing` (se encerrada).

## Regras Globais

1. Nenhuma automacao critica pula human gate.
2. Toda decisao automatica deve ser reversivel ou dry-run.
3. `docs/HANDOFF.md` continua a unica fonte de status atual.
4. `docs/human-gates.md` decide quando parar para aprovacao humana.
5. `docs/skill-workflow-matrix.md` define o papel de cada skill no fluxo.
6. Sentinel, Marshal, Oracle e Auditor fiscalizam; eles nao substituem o Andre.
7. `ABSTAIN` nunca vira `PASS` automaticamente.

## Papeis dos Agentes

| Agente | Papel no workflow | Limite |
|---|---|---|
| Sentinel | bloquear acoes inseguras em tempo real | nao aprova comando desconhecido |
| Marshal | detectar drift, pendencias e processo quebrado | nao decide prioridade sozinho |
| Oracle | propor calibragem por padrao observado | propostas sao dry-run ate aprovacao |
| Auditor | validar entrega contra evidencia | ABSTAIN exige review humano |
| Hermes/Vault | memoria, recall e aprendizado | nao decide acao critica |

## Papeis das Skills

Skills sao SOPs operacionais. Elas podem automatizar passos dentro de um
estado, mas devem respeitar:

- estado de entrada;
- estado de saida;
- human gate aplicavel;
- evidencia minima exigida.

A matriz completa esta em `docs/skill-workflow-matrix.md`.

## Human Gates

Human gates sao definidos em `docs/human-gates.md`. Regra curta:

```text
baixo risco + reversivel + dentro de escopo aprovado = automacao pode seguir
alto risco + irreversivel + producao/secrets/dados/arquitetura = Andre decide
```

## Evidencia Minima

Antes de declarar uma entrega pronta, reportar:

- escopo executado;
- arquivos alterados;
- comandos/testes rodados;
- resultado numerico;
- riscos remanescentes;
- pendencias ou ABSTAIN;
- decisao humana, quando houve.

## Dispatch Layer (2026-05-22)

Execution layer entre estado declarado e acao real. Operador roda
`legion workflow dispatch --skill X` quando state em ("approved", "executing");
SessionStart hook injeta no contexto da proxima Claude session.

Lifecycle do DispatchTarget:

```text
pending  ─(SessionStart hook le ledger)─►  injected
injected ─(operador roda dispatch-result)─► executed | failed
```

### dispatch_policy

| Policy | Comportamento |
|---|---|
| `auto-safe` (default) | Operador roda `dispatch` quando convem. Risk em SAFE_POLICY (leitura, analise, testes) — sem approval. Risk em CRITICAL_POLICY (producao, deploy, secrets) — exige `--approval-id`. |
| `manual` | Identico a auto-safe na pratica. Distincao reservada para futuro: manual deixa explicito "humano sempre na frente"; auto-safe pode ganhar auto-trigger em `transition --to executing`. |
| `gated` | Toda dispatch exige `--approval-id` independente de risk. Maxima cerimonia. |

### Anti-orchestrator preservado

Dispatch **NAO** decide qual skill chamar (Claude Code native faz). **NAO**
encadeia skills automaticamente (state machine governa, nao DSL). **NAO**
invoca agents externos (auto-injection via SessionStart hook eh o canal).

Spec: `docs/plans/2026-05-22-workflow-dispatch-design.md`.

## Nao Reintroduzir

Este contrato substitui a necessidade de:

- orchestrator custom;
- Ruflo bridge;
- skill-chains automaticas;
- dispatch regex-based.

Claude Code native selection + skills + hooks continuam o modelo ativo.
