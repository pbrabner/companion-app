## Estratégia de mock do Supabase (vi.mock)

Segui o pattern já estabelecido em `apps/web/src/shared/db/clients.test.ts` (T-005): `vi.mock('@supabase/ssr', ...)` no topo do arquivo de teste, retornando um `createServerClient` factory que devolve um stub com:

- `auth.getUser()` controlável via handle de closure (`getUserResult`)
- `from(...).select(...).eq(...).single()` controlável via handle (`profileResult`)

**Decisão chave — `vi.clearAllMocks()` em vez de `vi.resetAllMocks()`:** o pattern de T-005 usa `vi.resetModules()` + `vi.resetAllMocks()`, mas em T-007 a primeira tentativa com `resetAllMocks` quebrou os 5 testes seguintes ao primeiro porque o reset wipa as `mockImplementation` declaradas no `vi.mock()` factory (Vitest 2.x semantics). Trocando para `clearAllMocks` (que só limpa call history e preserva implementation), todos os 6 testes passam consistentemente.

Esta decisão é segura porque o middleware não tem state cross-test — cada `it` muda apenas os handles `getUserResult` / `profileResult` no `beforeEach`, que reseto explicitamente para defaults.

## Matcher do middleware Next.js

Configurei `config.matcher` excluindo paths estáticos e o health check:

```ts
matcher: ['/((?!_next/static|_next/image|favicon.ico|api/health).*)'];
```

Isto evita gastar uma chamada de `auth.getUser()` em cada asset do Next.js (overhead em prod) e mantém `api/health` aberto para o monitor externo (`vercel.json` provavelmente vai adicionar mais excluídos no futuro — out of scope T-007).

`/((?!...).*)` é o pattern recomendado pelo Next.js 15 (App Router) para "tudo, exceto X". Não usei `runtime: 'nodejs'` porque o Edge runtime é o default e funciona com `@supabase/ssr` (validado em T-005).

## Pattern de cookies (request.cookies + response.cookies)

Implementei `cookies.setAll` reconstruindo o `response` quando o Supabase precisa setar cookies (sessão refreshada):

```ts
setAll(cookiesToSet) {
  for (const { name, value } of cookiesToSet) {
    request.cookies.set(name, value);
  }
  response = NextResponse.next({ request });
  for (const { name, value, options } of cookiesToSet) {
    response.cookies.set(name, value, options);
  }
}
```

Razão: a doc oficial do `@supabase/ssr` recomenda esse padrão de "rewrite request, then re-issue response com cookies sincronizados". Isto garante que se o `getUser()` triggar um refresh do access_token, o novo cookie volta no header `Set-Cookie` da resposta. O `response` é declarado com `let` no escopo externo justamente para que a closure do `setAll` possa reescrevê-lo.

## Desvios e justificativas

**1. Não chequei `error` do `auth.getUser()` explicitamente.** O critério não pede tratamento de erro de auth. Se `getUser()` falhar com erro de rede, o destructuring `data: { user }` vai dar `user === undefined`, que cai no branch "não autenticado" → redirect para /login. Isto é seguro (fail-closed).

**2. Não chequei `error` do fetch de profile.** Mesma lógica fail-closed: se o fetch falhar, `profile` é `null`, `profile?.onboarded_at` é `undefined`, redirect para /onboarding. O usuário cai numa tela onde ele pode tentar de novo, em vez de entrar em /app sem onboarding. Marshal pode flaggar isto como concern em review — é decisão consciente, não esquecimento.

**3. Não usei `vi.resetModules()`.** T-005 usa para isolar imports dinâmicos com env vars diferentes; aqui não preciso, todos os tests usam o mesmo env e o mesmo módulo (`@/middleware`).
