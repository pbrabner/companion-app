## Modelo de autorização das 3 tabelas

| Tabela | Modelo | Políticas |
|---|---|---|
| `conversations` | owner-only direto via `user_id` | 4: `owner_select/insert/update/delete` (idênticas em formato às de T-004a) |
| `messages` | owner via **JOIN** em `conversations.user_id` (sem coluna `user_id` própria) | 4: `owner_select/insert/update/delete`, todas com subquery `conversation_id IN (SELECT id FROM conversations WHERE user_id = auth.uid())` |
| `safety_events` | SELECT owner-only + INSERT apenas via `service_role` | 2: `owner_select` + `service_only_insert TO service_role WITH CHECK (true)`. **Sem UPDATE/DELETE** (audit append-only) |

### Decisão `messages`: `IN (subquery)` vs `EXISTS`

Duas formas equivalentes considerados:

```sql
-- Forma A (escolhida)
USING (conversation_id IN (
  SELECT id FROM public.conversations WHERE user_id = auth.uid()
))

-- Forma B (equivalente)
USING (EXISTS (
  SELECT 1 FROM public.conversations c
   WHERE c.id = messages.conversation_id AND c.user_id = auth.uid()
))
```

**Escolhida:** Forma A. Justificativa: o critério de aceite do backlog é literalmente escrito como `select m.* from messages m where conversation_id in (select id from conversations where user_id = B)`. Manter a sintaxe da política em formato espelhado ao critério facilita auditoria visual. Performance é equivalente — o planner reescreve `IN (subquery)` em semi-join na maioria dos casos.

A subquery roda **sob o contexto de RLS do mesmo usuário autenticado**: o `SELECT id FROM conversations` interno aplica a política `owner_select` de `conversations`, então só retorna ids que o caller já pode ver. Isso é defesa em camadas: mesmo se a expressão fosse mal-lida, ela não vazaria conversations alheias porque a leitura é filtrada antes pela RLS de `conversations`.

### Decisão `safety_events`: sem UPDATE/DELETE policies

Seguindo o critério "audit append-only" do backlog:

- **Sem `owner_update`**: usuário não pode editar seu próprio evento de safety. Se a Server Action precisar corrigir um evento, ela roda como `service_role` e bypassa RLS.
- **Sem `owner_delete`**: mesma justificativa. Hard delete só por `service_role` (ou via cascade quando `auth.users` é deletado — `safety_events.user_id REFERENCES auth.users ON DELETE CASCADE`).
- **Sem `owner_insert`**: usuário comum não insere `safety_event` — a Server Action que detecta o evento de segurança é quem grava, sempre como `service_role`. Inclusive impede que um usuário forge um evento "limpo" pra si mesmo.

A política `service_only_insert TO service_role WITH CHECK (true)` é a única via de escrita autorizada pela camada de policy.

## Service_role e `rolbypassrls = t` — defesa em camadas

Verificação no DB local (executada via `docker exec supabase_db_companion-app psql`):

```
    rolname    | rolbypassrls
---------------+--------------
 authenticated | f
 anon          | f
 service_role  | t   ← bypassa RLS
 postgres      | t   ← bypassa RLS
```

**Implicação:** `service_role` ignora RLS por padrão. Logo, o INSERT em `safety_events` feito via cliente service_role passa **mesmo se a política `service_only_insert` não existisse**. Por que então criar a política?

**Justificativa de defesa em camadas:**
1. Se o flag `rolbypassrls` for revogado de `service_role` por alguma razão operacional (ex: hardening por compliance), a política ainda autoriza o caminho legítimo.
2. Se `safety_events` for futuramente exposta via uma view ou wrapper que re-aplique RLS (ex: para audit indireto), a política continua valendo.
3. Documenta no schema a intenção de autorização: leitor humano de `pg_policies` vê explicitamente "INSERT apenas por service_role". Sem a política, esse contrato fica implícito apenas no flag do role.

Esta nuance está documentada também no header da migration.

## Simulação de `service_role` em pgTAP

Pattern usado no teste (clause c — service_role inserts safety_event):

```sql
SET LOCAL ROLE service_role;
RESET "request.jwt.claim.sub";

SELECT lives_ok(
  $$ INSERT INTO public.safety_events (user_id, trigger_text, action_taken)
     VALUES ('<uuid>'::uuid, 'service insert', 'handoff') $$,
  'service_role can INSERT safety_event'
);
```

**Detalhe:** o `RESET "request.jwt.claim.sub"` antes do insert é uma boa prática higiênica — `service_role` não deve depender de JWT claim. O `SET LOCAL ROLE service_role` substitui o `authenticated` da seção anterior, e como `service_role` bypassa RLS, o insert passa direto. Como observado acima, ele também passaria pela política — verificado por construção no GREEN.

## Decisões de implementação

- **Reutilização do pattern de T-004a:** mesmo idioma de seed (`auth.users` rows mock antes de trocar role), mesmo idioma de impersonação (`SET LOCAL ROLE authenticated` + `SET LOCAL "request.jwt.claim.sub"`), mesmos UUIDs (`...000a` e `...000b`). Não foi extraído um helper compartilhado porque os dois testes ainda divergem em quais tabelas seedam — extração faz mais sentido se houver um terceiro teste de RLS.

- **Subquery idêntica nas 4 policies de `messages`:** poderia haver uma "policy `FOR ALL`" com a mesma subquery, evitando repetição. Mantive 4 policies separadas pelos mesmos motivos de T-004a (auditabilidade visual em `pg_policies`, expressividade do `cmd`). Verbosidade aceita.

- **`WITH CHECK` em `messages` insert também usa subquery:** confirmado que Postgres aceita subquery em `WITH CHECK` (não foi necessário recorrer a WebFetch — DB local validou pelo simples fato de a migration aplicar e o teste passar). Comportamento: ao inserir uma mensagem, o `WITH CHECK` re-executa a subquery contra `conversations` com a RLS do caller; se `conversation_id` da nova mensagem não estiver no conjunto visível, o INSERT é rejeitado com `42501`. Validado pelo assert "A cannot INSERT message into B's conversation".

- **`safety_events.service_only_insert WITH CHECK (true)`:** sem predicado nenhum no WITH CHECK porque service_role é confiável por definição na arquitetura (Architecture A2). A Server Action que invoca `service_role` é responsável por validar que o `user_id` que está gravando é o usuário correto. Pôr `WITH CHECK (auth.uid() = user_id)` aqui seria contraprodutivo — service_role normalmente não tem JWT claim sub, então `auth.uid()` retornaria NULL e o INSERT falharia.

- **Test plan = 31, não 28:** primeira tentativa usou `plan(28)` mas o arquivo continha 31 asserts (over-count na minha cabeça). pgTAP detectou e reportou "Bad plan. You planned 28 tests but ran 31" — corrigi para `plan(31)` antes do commit RED final.
