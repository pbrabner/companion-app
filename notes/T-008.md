Lição pra futuras migrations: ao DROP INDEX, fazer `grep -rn "<index_name>" supabase/tests/` antes do GREEN phase. Plan deveria ter previsto.

Re-validar no test novo é redundância controlada — não é bug, é re-cobertura explícita do gate ★ALTO no contexto da feature core.

## Reflexão sobre o fluxo

T-008 foi prova de fogo do fluxo Dev do Legion (Spec→Plan→Execute). Caminho:
1. PRD aprovado em validation-lab (idea_id=1, VIABLE T:9 M:8 D:7)
2. Brainstorming com 5 perguntas sequenciais (insights_jsonb defer, append-only, sem 1/dia, sem CHECK size, index unificado)
3. Spec v0.1 escrita
4. Erro detectado durante writing-plans: `journal_entries` já existia
5. Spec v0.2 corrigida (D-T008-9, D-T008-10 adicionadas) antes de codar
6. Plan escrito com SQL literal
7. TDD red → green → suite full → notes
8. 4 commits em main: chore (scaffold) → test (RED) → feat (GREEN) → docs (notes)

Aprendizados:
- **"Verificar antes de criar" não é opcional**. Brainstorming devia ter feito grep em migrations existentes antes de propor schema novo. Spec v0.1 falhou; v0.2 corrigida sem perda significativa porque writing-plans atuou como gate. Custo do erro: 1 retrabalho de spec (~15 min). Custo se tivesse passado: 1 migration órfã + retrabalho em T-009/T-010.
- **Code review subagent encontrou bug real** (`plan(15)` vs 16 assertions) que self-review do implementer não pegou. Validação do pipeline subagent-driven com 2-stage review.
- **DROP INDEX em test que assertia o nome antigo é bug latente predictable**. Plan poderia listar como Step 3.5 (não só rls_direct).
