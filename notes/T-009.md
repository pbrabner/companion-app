Spies em `console.log/info/warn/error`. Asserção: `JSON.stringify(spy.mock.calls).includes(SENTINEL) === false`. False positive estatisticamente impossível.

## Desvios da spec

### 1. Strengthening do privacy gate em runtime (issue importante encontrada por code review)

Plan original `route.ts` step 3.2 logava `err.message.slice(0, 64)` no catch do `chatStream`. Code reviewer da Task 3 detectou risco real: Anthropic SDK errors (e.g. `APIError`, validação 400) podem ecoar fragmentos do payload da request no `message` body, vazando user content em logs/observability. 64-char slice limita exposição mas não elimina.

**Fix aplicado (amend GREEN commit):** `error_code` agora usa `err.constructor.name` ('APIError', 'APIConnectionError') em vez de `err.message`. Comentário explicativo no código documenta o porquê. Sentinel test continua passando (sentinel >> 64 chars + estava em content, nunca em err.message), mas a regra "never logs content/body" agora vale absoluta.

Spec D-T009-11 mantida pelo espírito; implementação ficou mais estrita que o plan literal — gain de segurança sem custo.

### 2. TS strict-mode array indexing nos tests (issue minor herdada do RED phase)

Tests escritos no plan acessam `lines[0]` e `mock.calls[0][0]` direto, mas o `tsconfig.json` do Companion tem `noUncheckedIndexedAccess: true` (ou strict equivalent). 4 errors `TS2532` apareceram só ao rodar `pnpm typecheck` (não no Vitest runtime).

**Fix aplicado (amend GREEN commit):** `lines[0] ?? ''` e `mock.calls[0] ?? []` em 4 sites do `route.test.ts` (linhas 209, 297, 300, 381). Tests continuam falhando loud no runtime se o stream estiver vazio (JSON.parse('') throws), só silenciam o type-checker. Reviewer notou que a alternativa mais honesta seria `lines[0]!` after explicit length assertion — adotado o `?? ''` por ser menos cirúrgico.

Aprendizado: o plan deveria ter antecipado este detalhe e usado o pattern safe desde o RED phase.

### 3. Conta de testes: 12 cenários do plan → 11 it blocks reais

Plan declarou "12 cenários" no route.test.ts, mas algumas validações foram consolidadas em um mesmo `it` com sub-cases (e.g. `invalid_input` cobre missing-content E non-string em uma única função; `too_short` cobre <3 chars E whitespace-only). Resultado: 11 `it()` blocks que cobrem ~13 assertions distintas. Spec compliance review do RED phase confirmou que TODAS as 8 CAs (CA-T009-1..8) estão presentes via `it` específicos.

Sem perda de cobertura — só uma diferença de granularidade de empacotamento.

## Reflexão sobre o fluxo

T-009 foi a primeira task user-facing do Companion (primeiro endpoint público autenticado). Plan TDD bite-sized seguindo padrão T-008 (RED → GREEN → suite full → notes → push). 4 commits sequenciais em main + spec/plan commits prévios.

**Aprendizados:**
- Route Handler com stream é simples uma vez que `chatStream` async iterator existe (T-006 pagou a complexidade lá). Pattern `ReadableStream` controller + TextEncoder + try/finally é direto.
- Mock pattern com closed-over handles (espelhando `middleware.test.ts`) escala bem pra cenários diversos sem rebuild factory.
- Sentinel único pra privacy gate é técnica auditável de "prove by construction" — vale repetir em todos os endpoints futuros que tocam content sensível. Custo: 1 import (`randomUUID`) + 1 assertion via `JSON.stringify`.
- Code review apertou privacy gate ALÉM do que a spec literal pedia (constructor.name vs message slice). Habit de "spec é mínimo, qualidade é teto" funcionou.
- TDD discipline funciona: tests escritos antes pegaram bugs reais (TS strict, validation order edge cases).

**Erros que evitamos (vs T-008):**
- Sem "Verificar antes de criar" issue — não havia tabela existente pra checar (Route Handler é greenfield), mas verificamos `apps/web/src/app/api/` pra confirmar zero pre-existing routes. Mesmo cuidado.
- Sem `plan(N)` discrepância — pgTAP foi T-008 only.
