## Desvios em relação ao backlog (registrados conforme regra Diferença #5 do core)

### Desvio 1 — FKs para `tracks_catalog` adiadas para T-003b

O schema do EDD declara três colunas que referenciam `tracks_catalog(slug)`:

- `profiles.active_track text references tracks_catalog(slug)`
- `conversations.track_slug text references tracks_catalog(slug)`
- `track_progress.track_slug text not null references tracks_catalog(slug)`

A tabela `tracks_catalog` é criada em **T-003b**, que roda DEPOIS de T-003a (migration `0002_*.sql` aplica após `0001_*.sql`). Se T-003a tentasse criar essas FKs, o `CREATE TABLE` falharia — `tracks_catalog` ainda não existe.

**Resolução adotada nesta task:** as colunas são criadas com o tipo `text` (e `not null` onde EDD declara), **sem `REFERENCES tracks_catalog(slug)`**. T-003b tem a responsabilidade extra de adicionar as 3 FKs via `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY (...) REFERENCES tracks_catalog(slug)` ao final da sua migration, depois de criar `tracks_catalog` e popular os 3 slugs do seed.

**Nota para T-003b:** este desvio implica trabalho extra na T-003b — adicionar as 3 ALTER TABLE no final do `0002_support_schema.sql`. Já está logicamente no escopo de T-003b ("tabelas de suporte"), mas vale destacar para o Executor de T-003b.

### Desvio 2 — Test pgTAP usa `SET LOCAL session_replication_role = 'replica'`

As tabelas com `user_id uuid not null references auth.users(id)` exigem que o user_id inserido exista em `auth.users`. Em ambiente de teste pgTAP, o DB local começa vazio (`auth.users` sem linhas). Inserir uma linha completa em `auth.users` exige preencher vários campos não-nulos com defaults internos do GoTrue.

**Solução:** `SET LOCAL session_replication_role = 'replica'` no início do teste, dentro do BEGIN. Isso bypassa triggers e FK constraints durante a transação. A migration de produção ainda tem as FKs efetivas — o bypass só vale dentro daquele `BEGIN ... ROLLBACK` do teste.

Esta é a forma idiomática para testar comportamento de tabelas Supabase sem precisar simular Auth completo. Quando T-004a/b chegar e testar RLS, aí sim será necessário simular sessão autenticada (via `set_config('request.jwt.claim.sub', ...)`); mas para T-003a o foco é apenas estrutura + índice único.

## Decisões de implementação (não desvios — escolhas dentro do espaço de design)

- **`gen_random_uuid()`** como default das PKs uuid (PostgreSQL 13+ built-in via pgcrypto, habilitado por default no Supabase).
- **`on delete cascade`** preservado em todas as FKs para `auth.users` (conforme EDD), garantindo que apagar um auth user limpa todos os dados pessoais associados — alinhado com futuro requisito LGPD de "right to be forgotten" (registrado em Não-Funcionais a endereçar do Architecture).
- **CHECK constraint em `messages.role`** preservada (`role in ('user','assistant','system')`) para validação no banco, não só no app.
- **CHECK constraint em `checkins.mood`** preservada (`mood between 1 and 5`).
- **`patterns_json jsonb default '[]'::jsonb`** em `user_insights` — array vazio JSON, não `null`, para que UI possa fazer `patterns_json.length === 0` sem null-check.
- **Índice único `checkins_one_per_day`** usa expressão `(((created_at AT TIME ZONE 'UTC'))::date)`. **Divergência intencional do EDD**: o EDD escreve `(created_at::date)` literalmente, mas Postgres rejeita esse cast em índice porque `timestamptz::date` depende da session TimeZone (não é IMMUTABLE; SQLSTATE 42P17). Resolução: ancorar o bucket em UTC explicitamente. Semântica final: "um check-in por dia UTC por usuário". Conversão para timezone local fica a cargo da UI no momento de exibir histórico/streak. Esta divergência cabe no EDD ser atualizado em iteração futura (já listado em "Backlog Futuro → Conteúdo" no backlog).
