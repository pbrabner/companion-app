### Desvio 2 — Smoke do remoto Supabase pendente até credenciais chegarem

O critério de aceite tem duas cláusulas:

1. **Local:** `supabase start` + `supabase status` → URLs locais. **Esta cláusula é validada nesta execução.**
2. **Remoto:** `select 1` via supabase-js apontando para o remoto → `[{ "?column?": 1 }]`. **Esta cláusula está pendente** porque o owner não anexou as credenciais do projeto remoto (URL + anon key + service_role key) ao prompt desta sessão, embora a instrução tenha indicado que elas existiriam.

A cláusula remota é validada por **smoke manual** assim que o owner enviar as 3 strings. Procedimento previsto:

```bash
# (após owner colar as 3 strings em apps/web/.env.local)
cat > apps/web/scripts/smoke-remote.ts <<'EOF'
import 'dotenv/config';
import { createClient } from '@supabase/supabase-js';
const c = createClient(process.env.NEXT_PUBLIC_SUPABASE_URL!, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!);
const { data, error } = await c.rpc('select_one'); // ou .from('_dummy').select('1') quando houver tabela
console.log({ data, error });
EOF
pnpm tsx apps/web/scripts/smoke-remote.ts
# inspecionar output, depois:
rm apps/web/scripts/smoke-remote.ts
```

Em alternativa simples: rodar `supabase db remote status` ou similar contra o remoto após `supabase link` — também valida conectividade sem precisar de tabela de teste.

**Status atual:** ⬜ smoke remoto pendente até credenciais chegarem.

## Decisões de implementação (não desvios — escolhas dentro do espaço de design)

- **Cliente SSR via `@supabase/ssr`:** versão atual recomendada pelo Supabase para Next.js App Router (substituiu `@supabase/auth-helpers-nextjs`). Provê `createServerClient` que usa cookies do `next/headers` para sessão.
- **`createServerClient` exportada como factory** (não singleton): cada Server Action/Route Handler chama a factory com cookies frescas. Isolamento de contexto + RLS funciona corretamente.
- **`createBrowserClient` e `createServiceClient` ficam para T-005** — esta task implementa apenas `createServerClient` (escopo declarado pelo backlog T-002 é "configurar Supabase local + projeto remoto novo"; T-005 é quem entrega os 3 clientes formalmente).
- **Mock de `@supabase/ssr` em testes:** `vi.mock` retorna factory que devolve um stub com `.from()`. Permite validar a forma do código sem bater no remoto.
- **`.env.local` é o único lugar com valores reais.** `.env.example` tem placeholders. `.gitignore` (já criado em T-001) bloqueia `.env.local`.
