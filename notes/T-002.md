```bash

import 'dotenv/config';

EOF
pnpm tsx apps/web/scripts/smoke-remote.ts
# inspecionar output, depois:
rm apps/web/scripts/smoke-remote.ts
```

### Desvio 5 — `supabase start` bloqueado por I/O error do Docker Desktop containerd

Após 3 tentativas (incluindo restart limpo do Docker Desktop e wait de 30s), `npx supabase start` falha persistentemente com:

```
Error response from daemon: write /var/lib/desktop-containerd/daemon/io.containerd.metadata.v1.bolt/meta.db: input/output error
```

`docker pull public.ecr.aws/supabase/postgrest:v14.10` direto reproduz o mesmo erro, confirmando que **não é problema do supabase CLI** — é corrupção/lock no metadata.db do containerd embutido no Docker Desktop. Causas comuns:
- WSL2 backend não totalmente estável
- Disco cheio na partição que hospeda o `desktop-containerd`
- Bug conhecido do Docker Desktop (várias issues abertas no GitHub do Docker Desktop com este símbolo)

**Procedimento de mitigação esperado** (a executar pelo owner antes da próxima task que dependa de banco local):
1. `docker system df` — verificar uso de espaço.
2. Em Docker Desktop → Troubleshoot → "Reset to factory defaults" OU **manualmente apagar `/var/lib/desktop-containerd/`** dentro da VM WSL e reiniciar Docker Desktop.
3. Re-rodar `npx --prefix apps/web supabase start` para confirmar.

**Status do critério de aceite local:** parcialmente verificado.
- `supabase init` ✅ (config.toml criado, comprovando que o CLI está acessível e funcional)
- `supabase start` ⛔ (bloqueio externo de infraestrutura)
- `supabase status` ⛔ (depende do start)

**Por que ainda assim avançar com commit GREEN:** o código produzido (`server.ts`, `.env.example`, `config.toml`, README, deps) está completo, testado e correto. O bloqueio é ambiental e reversível pelo owner. Forçar reset destrutivo do Docker Desktop sem permissão violaria o princípio "Executando ações com cuidado" (alto blast radius para o ambiente de dev do owner). Próxima task que precisar de banco local (T-003a, migration) terá que confirmar que `supabase start` ficou OK depois da intervenção do owner.
