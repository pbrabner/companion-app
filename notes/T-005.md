## Pattern de `import 'server-only'` em `service.ts`

A regra "literal primeira linha" admite uma sutileza: TSDoc é **comentário**, não código executável; ESLint/TypeScript não tratam comentários como statements. Portanto a ordem correta é:

```typescript
/**
 * TSDoc header (cumpre a regra Legion validate_headers.py)
 * @module shared/db/service
 */
import 'server-only';

import { createClient } from '@supabase/supabase-js';
// ...
```

**TSDoc → blank line → `import 'server-only';` → blank line → outros imports → código.**

O teste `imports server-only as the very first import (defense in depth)` em `clients.test.ts` lê o arquivo via `fs.readFileSync`, **strippa o leading TSDoc block** com regex `/^\s*\/\*\*[\s\S]*?\*\/\s*/`, e assert que a primeira linha não-vazia depois é exatamente `import 'server-only';`. Isso valida o invariante semântico (server-only é o primeiro código executável) sem cair na armadilha de exigir literalmente "byte 0".

Esta abordagem alinha com a recomendação Next.js + permite cumprir a regra Legion (header obrigatório). Verificar o snippet exato no [docs Next.js — `server-only`](https://nextjs.org/docs/app/getting-started/server-and-client-components#preventing-environment-poisoning) confirma que o padrão idiomático é "import 'server-only' antes dos demais imports" — não obriga byte 0.

## Script `db:types`: wrapper Node.js, não shell direto

Primeira tentativa: `"db:types": "supabase gen types typescript --local > src/shared/db/types.ts"`.

**Problema:** o Supabase CLI emite "Using workdir ..." e "Connecting to db 5432" no stderr antes do conteúdo TypeScript válido em stdout. Se o terminal redireciona stderr também (ou usa um shell que mistura), o output fica corrompido na primeira linha. Na primeira execução desta task, o redirect simples `>` capturou apenas stdout (correto), mas re-rodar via `pnpm db:types` apagou meu TSDoc header (esperado — o regen sobrescreve).

**Solução:** criei `scripts/gen-db-types.mjs` que:
1. Roda `pnpm exec supabase gen types typescript --local` via `spawnSync` com stdin/stderr ignorados, stdout capturado.
2. Prepende um TSDoc header de "auto-generated, DO NOT EDIT BY HAND".
3. Escreve atomicamente em `src/shared/db/types.ts`.

Vantagem: regenerar via `pnpm db:types` sempre produz arquivo com header válido; `validate_headers.py` (regra Legion) não bloqueia commits.

Detalhe do `shell: true`: necessário no spawnSync porque `pnpm` no WSL/Linux requer shell PATH lookup (não é executável direto via Node child_process). Risk de injection é zero — todos os args são literais.

## Decisões de implementação

- **`<Database>` generic em todos os 3 clients:** browser.ts e service.ts usam `createBrowserClient<Database>` e `createClient<Database>`, espelhando o pattern de server.ts. Isto garante que `client.from('safety_events').insert(...)` valide tipos da Insert ao invés de aceitar `any`.

- **`auth: { autoRefreshToken: false, persistSession: false }` em service.ts:** service_role não tem sessão de usuário — auto-refresh quebraria. Idiomatic pattern do Supabase docs para "server-side admin client".

- **`readEnv(name)` duplicado em 3 arquivos:** poderia ser extraído para `shared/db/_env.ts`. **Decisão de não extrair:** o helper tem 7 linhas e duplicação reflete intenção (cada client lê suas próprias env vars; falha cedo se faltar). Refatorar quando aparecer um 4º consumer ou se o erro precisar ser instrumentado (Sentry/etc.).

- **`vi.mock('server-only', () => ({}))` no test:** `server-only` é um module sem exports (apenas tem efeito de build-time barrier). Em runtime de teste (Vitest, environment 'node'), simplesmente importar funciona — mas explicitar o mock evita warning sobre missing module em alguns setups Vite.

- **`vi.resetModules()` em `beforeEach`:** importante porque cada `it` usa `await import(...)` para que os mocks `vi.mock` sejam aplicados antes da resolução (módulos resolvidos uma vez ficam em cache do Vite).

- **Não implementei singleton para browser client:** docs Supabase recomendam singleton no client (uma única instância para o lifecycle do app), mas isso pode ser camada de cima (e.g., um React Context Provider em T-009/T-010). T-005 entrega só a factory — instanciar/cachar é decisão da camada de UI.
