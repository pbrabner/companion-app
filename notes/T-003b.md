## Esclarecimentos sobre fonte canônica do schema

### Sobre `safety_events` — seguir EDD, não o prompt

O prompt do owner desta sessão sugeriu colunas adicionais para `safety_events`:
`(id PK, user_id, conversation_id, message_id, trigger_text, action_taken, created_at)`. Anotando o "**colunas exatas conforme EDD**" do mesmo prompt, a fonte canônica é o EDD §Schema de Dados, que define:

```sql
safety_events (
  id            uuid primary key default gen_random_uuid(),
  user_id       uuid not null references auth.users(id) on delete cascade,
  trigger_text  text not null,
  classifier    text,
  action_taken  text not null,
  created_at    timestamptz default now()
);
```

Sem `conversation_id` nem `message_id`. Esta task segue **EDD literalmente**. Se essas colunas vierem a fazer sentido na implementação do Safety Layer (T-018), entram via migration futura — não nesta. Razão: EDD/Architecture/Backlog são contratos canônicos; expandir schema sob suspeita aqui geraria divergência silenciosa que o Architect futuro precisaria reconciliar.

### Sobre `usage_counters` — esquema derivado de Architecture A5 + T-019

Architecture §"Decisões de Stack" A5 lista as colunas-chave: `chat_msgs_day`, `chat_msgs_hour`, `last_warning_at`, com PK `(user_id, day_bucket)`. O backlog T-019 adiciona "Bucket horário/diário em UTC". Esquema final adotado:

```sql
usage_counters (
  user_id          uuid not null references auth.users(id) on delete cascade,
  day_bucket       date not null,                            -- UTC calendar day
  hour_bucket      timestamptz not null,                     -- date_trunc('hour', now() AT TIME ZONE 'UTC')
  chat_msgs_day    int  not null default 0,
  chat_msgs_hour   int  not null default 0,
  last_warning_at  timestamptz,
  updated_at       timestamptz not null default now(),
  primary key (user_id, day_bucket)
);
```

**Por que `hour_bucket` é coluna, não parte do PK:** uma única linha por user-dia rastreia tanto janela diária quanto horária. A lógica do `incrementCounters` (T-019) faz upsert na linha do dia e reseta `chat_msgs_hour` quando `hour_bucket` mudou. Manter PK com 3 colunas explodiria o número de linhas (até 24/dia/usuário) sem ganho — o estado horário relevante é sempre só "agora". Coerente com o objetivo do contador (rate limit ativo, não histórico longitudinal).

## Decisões de implementação

- **Seed das 3 trilhas via `INSERT ... ON CONFLICT (slug) DO NOTHING`** — idempotente; rodar a migration em DB já populado não duplica nem falha.
- **Títulos e descrições das trilhas em pt-BR** — produto é PT-BR-only no MVP (Backlog Futuro lista i18n para depois). Texto editorial **não final** (placeholder funcional); produção real é responsabilidade do owner conforme Decisão §2 do Orchestrator. `steps_total=3` para todas, alinhado com o número de etapas placeholder de T-016.
- **Nomes de constraint explícitos** nas 3 ALTER TABLE ADD CONSTRAINT (e.g., `profiles_active_track_fkey`) para tornar o catálogo de FKs legível e referenciável em migrations futuras (drop/replace).
- **FK de `track_progress.track_slug` é `NOT NULL`** (já era na migration 0001) — apenas adicionada a constraint REFERENCES sobre coluna existente.
- **`safety_events` recebe FK para `auth.users` direta** com `ON DELETE CASCADE`, conforme EDD. Embora EDD diga "auditoria, nunca apagar", o cascade preserva consistência referencial; política de retenção/anonimização no LGPD vai além do schema (registrado em Não-Funcionais a endereçar do Architecture).
