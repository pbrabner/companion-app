## Escopo: 6 tabelas, mas `profiles` é especial

As tabelas com identificador direto do dono são 6, mas elas não usam o **mesmo nome de coluna**:

| Tabela            | Coluna-chave para `auth.uid()` | Observação |
|-------------------|--------------------------------|------------|
| `profiles`        | **`id`**                       | PK referencia `auth.users(id)` direto — não há `user_id` separado |
| `checkins`        | `user_id`                      | |
| `journal_entries` | `user_id`                      | |
| `track_progress`  | `user_id`                      | PK composta (user_id, track_slug) |
| `user_insights`   | `user_id`                      | `user_id` é a PK |
| `usage_counters`  | `user_id`                      | PK composta (user_id, day_bucket) |

**Decisão para `profiles`:** as 4 políticas (`owner_select/insert/update/delete`) usam predicado `auth.uid() = id` em vez de `auth.uid() = user_id`. Justificativa: o EDD modela `profiles` como extensão 1-1 de `auth.users` — adicionar uma coluna redundante `user_id` violaria normalização e introduziria drift entre `id` e `user_id`. O comportamento RLS é idêntico (usuário vê / muta apenas a própria linha), apenas o nome da coluna difere.

**Decisão para `usage_counters` e `track_progress` (PK composta):** mesmo set de 4 políticas que as outras. PK composta não muda o modelo de autorização — `auth.uid() = user_id` é suficiente porque `user_id` é parte da PK em ambos os casos, garantindo que cada linha tem um único dono lógico.

## Técnica para simular `auth.uid()` em pgTAP

Investigação no DB local mostrou que `auth.uid()` é definida assim (relevante porque a documentação online fica defasada vs. o que o Supabase CLI realmente provisiona):

```sql
CREATE OR REPLACE FUNCTION auth.uid()
  RETURNS uuid LANGUAGE sql STABLE AS $function$
  SELECT coalesce(
    nullif(current_setting('request.jwt.claim.sub', true), ''),
    (nullif(current_setting('request.jwt.claims', true), '')::jsonb ->> 'sub')
  )::uuid
$function$;
```

A função lê o GUC `request.jwt.claim.sub` (formato singular usado pelo PostgREST/GoTrue do Supabase). Logo, simular auth no teste é direto: `SET LOCAL "request.jwt.claim.sub" TO '<uuid>'`.

**Detalhe crítico:** `SET LOCAL "request.jwt.claim.sub"` sozinho **não basta**. O role default no `supabase test db` é `postgres`, que tem `rolbypassrls = true` — RLS é simplesmente ignorada. É preciso também `SET LOCAL ROLE authenticated` (que tem `rolbypassrls = false`) para que as políticas sejam efetivamente aplicadas.

**Verificação no DB local:**

```
       rolname       | rolbypassrls
---------------------+--------------
 postgres            | t
 supabase_admin      | t
 service_role        | t
 authenticated       | f   ← este é o que o teste usa
 anon                | f
 supabase_auth_admin | f
```

`authenticated` já recebe os GRANTs default do Supabase em todas as tabelas `public.*` (verificado via `information_schema.role_table_grants` antes de escrever o teste), então a única barreira que resta para o usuário simulado é justamente a RLS — exatamente o comportamento que queremos validar.

### Setup das `auth.users` rows

A FK `user_id REFERENCES auth.users(id) ON DELETE CASCADE` exige que os UUIDs do teste existam em `auth.users`. Em T-003a usamos `SET LOCAL session_replication_role = 'replica'` para bypassar FKs, mas esse comando exige superuser — incompatível com `SET ROLE authenticated`.

**Solução adotada:** criar as 2 rows mock em `auth.users` **antes** do `SET LOCAL ROLE authenticated`. Apenas `id`, `is_sso_user` e `is_anonymous` são NOT NULL (todos os demais campos são nullable ou têm default), o que torna o `INSERT` mínimo trivial:

```sql
INSERT INTO auth.users (id, is_sso_user, is_anonymous)
VALUES
  ('00000000-0000-0000-0000-00000000000a', false, false),
  ('00000000-0000-0000-0000-00000000000b', false, false);
```

Isso roda como `postgres` (default), bypassando RLS para o seed. Depois trocamos o role para `authenticated` e impersonamos via JWT claim. O `BEGIN ... ROLLBACK` da pgTAP garante que essas rows mock não persistem entre tests.

### Limitação descoberta: CTEs com data-modifying não aceitam aninhamento

Tentativa inicial usava o pattern:

```sql
SELECT is(
  (WITH upd AS (UPDATE ... RETURNING 1) SELECT count(*) FROM upd),
  0,
  'A UPDATE of B affects 0 rows'
);
```

Postgres rejeitou: `ERROR: WITH clause containing a data-modifying statement must be at the top level`. Isto é um limite estrutural do parser — CTEs modificadoras só rodam no nível de query.

**Solução adotada:** executar os UPDATEs/DELETEs diretamente como statements (não dentro de subquery), depois `RESET ROLE` e validar **estado pós-tentativa** das rows de B (display_name, body, mood ainda no valor original; rows ainda existem). Esta abordagem é até mais robusta porque mede o efeito observável (B's data não foi alterada) em vez do mecanismo interno (rows afetadas pelo UPDATE) — alinhado com o princípio "RED semântico" das recalibrações herdadas.

## Decisões de implementação

- **Nomes de policy idênticos entre tabelas** (`owner_select`, `owner_insert`, `owner_update`, `owner_delete`): policies em PostgreSQL são namespaced por tabela, então não há colisão. Padronizar o naming facilita auditoria visual em `pg_policies` e em PRs de migration futuros que toquem o mesmo modelo (ex: T-004b).
- **`UPDATE` com `USING` E `WITH CHECK`**: `USING` filtra quais linhas estão visíveis para a operação (a USING USING expression); `WITH CHECK` impede que o usuário re-atribua o row a outro `user_id`/`id` no SET. Sem `WITH CHECK`, A poderia, em teoria, mudar `user_id` da própria linha para B e exfiltrar; com `WITH CHECK = (auth.uid() = user_id)`, esse caminho fica fechado.
- **`DELETE` só com `USING`**: não há SET em DELETE, não há necessidade de `WITH CHECK`.
- **Sem `FOR ALL`**: separar 4 políticas (vs. uma única `FOR ALL`) é mais verboso mas mais auditável. Cada política aparece nominalmente em `pg_policies`, e `pg_policies.cmd` indica claramente qual operação cada uma cobre.
