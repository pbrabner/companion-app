## Decisão central — `classifyHaiku` sem structured output nativo

A SDK Anthropic não expõe um modo "structured output" análogo ao da OpenAI (`response_format: json_schema`). A spec da task explicita o contorno: instruir o modelo via system prompt a responder **JSON puro** e fazer `JSON.parse` do primeiro bloco de texto.

Decisões implementadas:

1. **System prompt determinístico:** literal `"You are a strict JSON classifier. Reply with raw JSON only — no prose, no markdown, no code fences. The JSON must conform to this schema: <schemaDescription>"`. O `schemaDescription` é `JSON.stringify(args.schema)` — descritivo, não validador. Suficiente para guiar Haiku a emitir o shape correto.
2. **Parsing minimalista:** percorre `response.content`, pega o primeiro bloco `type === 'text'`, faz `JSON.parse(block.text)`. Se nenhum bloco de texto, joga `Error('classifyHaiku: model response did not contain a text block')`. Não tentamos "limpar" cercas markdown — o system prompt já proíbe; se vier sujo, o parse falha cedo (princípio fail-fast).
3. **Validação de shape:** **não** implementada nesta camada. O genérico `<T>` narrows o tipo no call site, mas não há `zod`/`io-ts` no critério de aceite. Caller que receba input não-confiável deve validar no boundary (ex: rota `/api/safety` em T-018 vai precisar de validação runtime; criar nessa camada quando for necessário, não antes).
4. **Sem retry no helper:** o critério de aceite é binário (mock retorna JSON válido → função retorna objeto). Retry e fallback são preocupações de uma camada acima (rate-limit handling, timeout — possivelmente em uma future task de "AI infra hardening").

Esta abordagem fica explicitamente atrás do `structured output` nativo em garantia de tipo, mas é o que a SDK atual permite. Quando/se a Anthropic SDK introduzir tool-use forçado para JSON (já existe um pattern com `tools` + `tool_choice: { type: 'tool', name: ... }`), considerar migrar.

## Decisão — `chatStream` consome `messages.stream` e re-emite só `text_delta`

A SDK Anthropic em modo streaming entrega um async iterator de **eventos** discretos (`message_start`, `content_block_start`, `content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`, `ping`). O consumer típico só quer o texto incremental para renderizar token-by-token na UI.

`chatStream` filtra: para cada evento, se `event.type === 'content_block_delta' && event.delta.type === 'text_delta'`, yield `event.delta.text`. Outros eventos (start/stop, message metadata, pings) são ignorados.

**Trade-off conhecido:** essa filtragem descarta `message_delta` events que carregam `usage.output_tokens` final e `stop_reason`. Quando precisarmos disso (telemetria de tokens, métricas de custo), o helper terá que evoluir — provavelmente para um shape `AsyncIterable<{ type: 'text', text: string } | { type: 'done', usage: {...} }>`. Por ora, T-006 entrega só strings, alinhado ao critério de aceite literal `["a","b","c"]`.

**Type-cast intencional:** `for await (const event of stream as AsyncIterable<unknown>)` — o tipo retornado por `messages.stream` é uma classe `MessageStream` que herda de `EventEmitter` E é async-iterable. Para evitar acoplamento ao tipo concreto da SDK (que muda entre versions), tratei como `AsyncIterable<unknown>` e fiz narrowing via `evt.type === 'content_block_delta' && evt.delta?.type === 'text_delta'`. Defensivo contra evolução breakage da SDK.

## Decisão — instanciar o client a cada chamada (não singleton de módulo)

Implementação atual:

```ts
function createClient(): Anthropic {
  const apiKey = process.env.ANTHROPIC_API_KEY;
  return new Anthropic({ apiKey });
}
```

Cada `chatStream` ou `classifyHaiku` cria um `new Anthropic(...)` no momento da chamada. Razões:

1. **Env-var validation lazy:** se `ANTHROPIC_API_KEY` faltar, o erro acontece no call site (production runtime), não no module import (que quebraria carregamento de qualquer rota que importe transitivamente o módulo, incluindo páginas estáticas que nem usam IA).
2. **Compatibilidade com testes:** `vi.mock('@anthropic-ai/sdk')` é resolvido no module load. Se eu cacheasse uma instância no top-level do módulo, ela seria criada antes do mock estar wired — falha de timing. Lazy instantiation contorna isso sem cerimônia.
3. **Cost de instanciar:** `new Anthropic({ apiKey })` é trivial — só constrói um wrapper que segura a key e mantém um pool de fetch interno. Não é uma conexão real. Re-instanciar a cada chamada não tem overhead mensurável.

Se em algum momento a SDK começar a manter caches importantes na instância (atualmente não), trocamos para um lazy singleton com guard. Por ora, simplicidade vence.
